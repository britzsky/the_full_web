# AGENTS.md

## 역할
이 프로젝트에서 에이전트는 React, Java, Spring Boot, MySQL을 함께 다루는 풀스택 개발자처럼 작업합니다.

기술 스택:
- Frontend: React
- Backend: Java, Spring Boot
- Database: MySQL

에이전트는 먼저 현재 수정 대상이 어떤 영역인지 판단한 뒤, 해당 영역의 규칙을 우선 적용합니다.

---

## 적용 기준

### 프론트엔드 규칙을 우선 적용하는 경우
아래와 같은 파일 또는 코드 구간을 수정할 때는 프론트엔드 규칙을 우선 적용합니다.
- `.js`
- `.jsx`
- `.ts`
- `.tsx`
- `.css`
- `.scss`
- 프론트엔드 컴포넌트 파일
- 프론트엔드 페이지 파일
- 프론트엔드 상태관리 파일
- 프론트엔드 API 호출 유틸 파일

### 백엔드 규칙을 우선 적용하는 경우
아래와 같은 파일 또는 코드 구간을 수정할 때는 백엔드 규칙을 우선 적용합니다.
- `.java`
- Spring Controller
- Spring Service
- Spring Repository
- DTO / VO / Entity 관련 파일
- 백엔드 설정 파일

### 데이터베이스 규칙을 우선 적용하는 경우
아래와 같은 파일 또는 코드 구간을 수정할 때는 데이터베이스 규칙을 우선 적용합니다.
- `.sql`
- MyBatis Mapper XML
- 프로시저
- 함수
- 트리거
- 복잡한 조회 / 집계 / 수정 쿼리

### 여러 영역을 함께 수정하는 경우
프론트엔드, 백엔드, 데이터베이스를 함께 수정하는 경우에는 아래 기준을 따릅니다.
- 공통 규칙을 먼저 적용합니다.
- 각 파일별로 해당 영역의 규칙을 따릅니다.
- 한 영역의 수정 때문에 다른 영역의 기존 계약을 임의로 바꾸지 않습니다.
- API 요청값, 응답값, 쿼리 결과 의미는 연결된 영역 전체를 고려해 신중히 수정합니다.

## 최우선 규칙
- 기존 로직을 함부로 변경하지 않습니다.
- 원래 동작을 최대한 유지합니다.
- 필요한 부분만 최소 수정합니다.
- 모든 주석과 설명은 한글로 작성합니다.
- 주석은 수정 사실보다 코드의 역할과 구조를 설명하는 데 집중합니다.
- 요청하지 않은 대규모 리팩토링, 전면 포맷팅 변경, 파일 구조 변경은 하지 않습니다.
- 작업 전 현재 수정 대상이 프론트엔드인지, 백엔드인지, 데이터베이스인지 먼저 판단합니다.
- 판단한 영역의 규칙을 우선 적용하되, 다른 영역에 미치는 영향도 함께 고려합니다.

## 공통 작업 원칙

### 1. 기존 비즈니스 로직과 동작 흐름을 우선 보존합니다.
- 이미 동작 중인 계산식, 조건문, API 흐름, 데이터 처리 방식은 우선 유지합니다.
- 꼭 필요한 수정만 최소 범위로 반영합니다.

### 2. 기존 코드 스타일을 최대한 유지합니다.
- 들여쓰기
- 함수 작성 방식
- 컴포넌트 구조
- 서비스 / 컨트롤러 패턴
- SQL 작성 스타일

위 항목은 기존 프로젝트 스타일을 우선 따릅니다.

### 3. 영향 범위를 먼저 파악하고 수정합니다.
- 프론트 수정 시 백엔드 API 영향 여부를 고려합니다.
- 백엔드 수정 시 프론트 화면 및 응답 구조 영향을 고려합니다.
- SQL 수정 시 운영 데이터와 계산 결과 영향 여부를 고려합니다.

### 4. 불필요한 대규모 변경은 하지 않습니다.
- 요청 없는 전역 네이밍 수정 금지
- 요청 없는 전체 구조 개편 금지
- 요청 없는 import 정리만을 위한 대량 수정 금지
- 요청 없는 파일 전체 포맷팅 금지

### 5. 수정 전에 현재 코드의 역할을 먼저 파악합니다.
- 이 파일이 어떤 역할인지 파악합니다.
- 이 함수나 메서드가 어디서 호출되는지 확인합니다.
- 이 수정이 다른 화면, 다른 API, 다른 쿼리에 영향을 주는지 확인합니다.

## 주석 작성 원칙

### 1. 설명형 주석을 우선합니다.
주석은 아래 내용을 설명하는 데 집중합니다.
- 이 메서드가 무엇을 하는지
- 이 함수가 어떤 흐름에서 호출되는지
- 이 변수나 상태값이 무엇을 의미하는지
- 이 div, section, component가 화면에서 어떤 역할인지
- 이 API가 어떤 요청을 처리하는지
- 이 쿼리가 어떤 데이터를 조회, 저장, 계산하는지

### 2. 관리형 주석은 최소화합니다.
아래 같은 주석은 꼭 필요한 경우가 아니면 남기지 않습니다.
- 기존 로직 유지
- 조건만 추가
- 응답 구조 유지
- 운영 영향 최소화

이런 표현보다 실제 코드의 역할을 설명하는 주석을 우선합니다.

### 3. 주석은 짧고 바로 이해되게 작성합니다.
처음 보는 사람도 빠르게 구조를 파악할 수 있어야 합니다.

좋은 예시:
- 승인 대기 사용자 목록을 조회하는 메서드
- 선택한 월의 입금 내역을 저장하는 API 호출 함수
- 검색 조건 입력 영역
- 상세 팝업 상단 제목 영역
- 거래처별 월 마감 데이터를 저장하는 서비스 메서드
- 전월 미수 잔액을 가져오는 서브쿼리

지양할 예시:
- 기존 코드 유지
- 영향도를 줄이기 위한 처리
- 수정된 부분
- div 시작
- 버튼 영역
- count 증가

## 프론트엔드 작업 규칙

### 적용 대상
- React 컴포넌트
- 페이지 컴포넌트
- hooks
- 상태관리 코드
- 프론트엔드 API 호출 코드
- 스타일 파일

### 작업 원칙
1. 기존 화면 동작을 우선 유지합니다.
2. UI 수정 시 기존 이벤트 흐름과 상태 관리 로직을 함부로 바꾸지 않습니다.
3. props, state, useEffect, API 호출 흐름은 영향 범위를 확인한 뒤 최소 수정합니다.
4. 재사용 중인 컴포넌트는 공통 영향도를 먼저 고려합니다.
5. JSX 구조가 복잡하면 화면 영역 설명 주석을 한글로 추가합니다.
6. 이벤트 함수는 무엇을 위한 동작인지 드러나게 작성합니다.

### 프론트엔드 주석 기준
- 이 함수가 어떤 사용자 동작을 처리하는지 설명합니다.
- 이 state가 어떤 화면 상태를 관리하는지 설명합니다.
- 이 div나 section이 화면에서 어떤 영역인지 설명합니다.
- 반복 렌더링 블록이나 조건부 렌더링 블록은 어떤 데이터를 보여주는지 설명합니다.

### 프론트엔드 주석 예시

```jsx
{/* 검색 조건 입력 영역 */}
<div className="search-area">

{/* 승인 대기 사용자 목록 영역 */}
<div className="pending-user-list">

{/* 상세 팝업 상단 제목 및 닫기 버튼 영역 */}
<div className="modal-header">
```

```js

const fetchMonthlySettlement = async () => {};
```

## 백엔드 작업 규칙

### 적용 대상
- Controller
- Service
- Repository
- DTO / VO / Entity
- 비즈니스 로직 메서드
- 예외 처리 로직

### 작업 원칙
1. 기존 비즈니스 로직과 서비스 흐름을 우선 보존합니다.
2. Controller, Service, Mapper, Repository 수정 시 호출 관계를 깨지 않도록 합니다.
3. Request / Response 구조는 프론트엔드 영향 여부를 고려하여 신중히 수정합니다.
4. 예외 처리 추가 시 기존 정상 흐름을 방해하지 않도록 합니다.
5. 단순 getter / setter 수준에는 불필요한 주석을 달지 않습니다.
6. 복잡한 서비스 로직은 계산 목적이나 처리 대상을 중심으로 설명합니다.

### 백엔드 주석 기준
- 이 API가 어떤 요청을 처리하는지 설명합니다.
- 이 서비스 메서드가 어떤 데이터를 저장하거나 계산하는지 설명합니다.
- 이 로직이 어떤 조건에서 호출되는지 설명합니다.
- 이 응답값이 프론트에 어떤 형태로 전달되는지 필요한 경우 설명합니다.

### 백엔드 주석 예시

```java
// 승인 요청 상태의 사용자 목록을 조회하는 API
public ResponseEntity<?> getApprovalRequestList() {}

// 거래처별 월 마감 데이터를 저장하는 서비스 메서드
public int saveDeadlineBalance(Map<String, Object> param) {}

// 입금 내역을 기준으로 미수 금액을 다시 계산하는 로직
public void recalculateDepositBalance(String accountId, int year, int month) {}
```

## 데이터베이스 작업 규칙

### 적용 대상
- SQL 파일
- MyBatis Mapper XML
- 프로시저
- 함수
- 트리거
- 복잡한 조회 / 집계 / 수정 쿼리

### 작업 원칙
1. 기존 테이블 구조와 운영 데이터에 영향을 줄 수 있는 변경은 매우 신중히 수행합니다.
2. 운영 중인 컬럼의 의미를 임의로 바꾸지 않습니다.
3. 기존 쿼리의 결과 의미가 달라지지 않도록 합니다.
4. UPDATE, DELETE, ALTER 문은 영향 범위를 먼저 고려합니다.
5. JOIN, 서브쿼리, 집계 로직은 기존 계산 방식 유지를 우선합니다.

### 데이터베이스 주석 기준
- 이 쿼리가 어떤 데이터를 조회하는지 설명합니다.
- 이 서브쿼리가 어떤 기준값을 가져오는지 설명합니다.
- 이 집계식이 무엇을 계산하는지 설명합니다.
- 이 UPDATE 문이 어떤 값을 어떤 기준으로 반영하는지 설명합니다.

### 데이터베이스 주석 예시

```sql
-- 거래처별 당월 입금 합계를 조회
SELECT ...

-- 전월 미수 잔액을 가져오는 서브쿼리
SELECT ...

-- 환불 데이터를 제외한 실제 입금 금액 합계 계산
SUM(...)
```

## 수정 방식

작업할 때는 아래 순서를 따릅니다.

1. 기존 코드의 역할과 흐름을 먼저 파악합니다.
2. 수정 범위를 최소화합니다.
3. 기존 동작이 유지되는 방향으로 구현합니다.
4. 주석이 필요하면 이 코드가 무엇을 하는지 설명합니다.
5. 결과를 제시할 때는 아래 형식을 우선합니다.

### 권장 응답 형식
- 변경 목적
- 영향 범위
- 수정 내용
- 주의사항

예시:
- 변경 목적: 승인 대기 사용자 필터 조건 보완
- 영향 범위: 사용자 승인 목록 조회 API와 화면 목록 표시
- 수정 내용: 기존 조회 흐름은 유지하고 승인 대기 조건만 보완
- 주의사항: 기존 응답 구조와 프론트 렌더링 방식은 변경하지 않음

- 코드만 봐도 알 수 있는 기계적 주석 반복
